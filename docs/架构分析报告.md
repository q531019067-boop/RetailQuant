# RetailQuant 架构分析报告

> 审计日期：2026-06-22 | 基于 `sim-trading` 分支 | 对应大纲 v2026-06-20

---

## 一、总体评价

架构整体合理，分层清晰，核心设计模式选择得当。**12 个策略**全部注册成功，策略注册中心、数据源池、回测引擎三大核心模块质量较高。存在少量不一致和冗余，但**没有严重的架构缺陷**，属于可渐进修复的范围。

---

## 二、亮点（值得保持的设计）

### 2.1 策略注册中心 — 简洁高效

`rquant/strategy/registry.py` 的 `@register` 装饰器 + 子模块导入触发注册，是整项目最优雅的设计：

```python
@register
class MyStrategy:
    name = "MyStrategy"
    category = "xxx"
    def signal_buy(...): ...
    def signal_sell(...): ...
```

- 新策略只需在对应子包创建文件 + 在 `__init__.py` 加一行 `from .xxx import ...  # noqa: F401` 即可接入
- `scan_stock()` / `scan_category()` / `scan_sell()` 三个统一入口覆盖所有调用场景
- 异常包裹完善：单个策略报错不影响其他策略

### 2.2 数据源池 — Protocol + 健康度路由

`rquant/data_source/pool.py` 的 `KlineSource` / `QuoteSource` 协议 + `DataSourcePool` 优先级路由设计良好：

- 新数据源只需实现 Protocol 并注册，业务代码零改动
- 健康度跟踪 + 自动 failover 保证了外部接口不稳定时的系统韧性
- `to_dataframe()` 便捷封装隔离了原始行→DataFrame 的转换逻辑

### 2.3 回测引擎 — 真实约束完整

`rquant/backtest/engine.py` 对 A 股 T+1、整数手、佣金、印花税、滑点的建模是严谨的：

- `PositionLot` + FIFO 移除表达了 T+1 可卖状态
- `BrokerConfig` 将费率参数集中管理
- `BacktestResult` 的 `summary()` 方法清晰分离了简要指标和详细数据

### 2.4 SQLite 存储层 — WAL + 线程安全

`rquant/data_source/db.py` 的 thread-local 连接 + WAL 模式是 Flask/waitress 多线程环境下的正确做法。`get_conn()` 的健康检查（`SELECT 1` + 重建连接）处理了跨请求的连接失效问题。

### 2.5 Parquet 存储 — 独立、清晰

`rquant/data_source/parquet_store.py` 完全独立于 SQLite，不耦合现有代码。显式 Schema、`replace`/`append` 双模式、空 DataFrame 安全返回，设计完整。

## 三、问题清单

### 🔴 高优先级（建议修复）

#### 3.1 `portfolio.py` 与 `funds.py` 多用户设计不一致

| 模块 | 多用户支持 |
|------|-----------|
| `rquant/business/funds.py` | ✅ 每个用户独立 `funds.json`（走 `get_user_data_dir()`） |
| `rquant/business/user.py` | ✅ 支持 local + sim 多用户 |
| `rquant/business/portfolio.py` | ❌ 全局 `data/portfolio.json`，不区分用户 |

`portfolio.py` 头部注释写"不分多用户（单实例）"，但 `funds.py` 和 `user.py` 都已实现多用户隔离。买卖路由需要同时更新 `portfolio.py` 和 `funds.py`，当前两者数据目录不一致会导致跨模块状态错乱。

**建议：** `portfolio.py` 改为按 `user_id` 读写 `{user_dir}/portfolio.json`，与 `funds.py` 保持一致。

#### 3.2 `DataSourcePool` 模块级硬编码导入 Sina 源

```python
# rquant/data_source/pool.py 第 14 行
from .sina import SinaKlineSource, SinaQuoteSource
```

以及 `__init__` 中硬编码 `[SinaKlineSource()]`。这导致：
- 导入 `pool` 模块时必须能导入 `sina` 模块
- 加新源仍需改 `pool.py` 的 `__init__`

**建议：** `DataSourcePool.__init__` 初始化为空列表，由启动代码显式注册源（如在 `app_factory.py` 中 `pool.add_kline(SinaKlineSource())`），实现真正的可替换。

### 🟡 中优先级（建议优化）

#### 3.3 `workflow.py` 绕过注册中心直接导入策略类

```python
# rquant/research/workflow.py 第 23 行
from rquant.strategy.factor.multi_factor import MultiFactor

# 第 27 行
DEFAULT_STRATEGIES = ("MovingAverageCross", "RsiMeanReversion", "GridMartingale", "MultiFactor")
```

`DEFAULT_STRATEGIES` 写死了策略名列表，且直接 `import MultiFactor` 类而非通过 `rquant.strategy.get()`。如果策略改名或新增，这两处都需手动同步。

**建议：** 使用 `rquant.strategy.get(name)` 按名获取策略，`DEFAULT_STRATEGIES` 改为从注册中心动态获取（如按 category 过滤）。

#### 3.4 `data_source/__init__.py` 导出膨胀

当前 `__all__` 导出了 18 个符号，包含低层 API：

```python
from .db import (
    close_thread_conn, executemany,
    execute as db_execute,  # 重命名导出
    get_conn, meta_get, meta_set, query_all, query_one,
)
from .mq import Mq, mq, start_mq, stop_mq
from .quote_cache import QuoteCache, quote_cache
```

`close_thread_conn`、`executemany`、`get_conn` 等是 `db.py` 的内部实现细节，不应暴露在 `data_source` 包的公共接口中。`mq` 和 `quote_cache` 是否属于"数据源"层也值得商榷——它们更像是基础设施。

**建议：** 仅导出 `pool`、`DataSourcePool`、`KlineSource`、`QuoteSource`、`SinaKlineSource`、`SinaQuoteSource`、`CACHE_DIR`。`db`、`mq`、`quote_cache`、`parquet_store`、`eastmoney` 由业务层按需显式导入，不通过 `__init__.py` 中转。

#### 3.5 `rquant/business/__init__.py` 遗漏导出

```python
# 当前
from . import board, data, funds, market, portfolio, user
__all__ = ["board", "data", "funds", "market", "portfolio", "user"]
```

遗漏了 `pool_store` 和 `system` 两个模块。虽然外部通过 `data.get_pool()` 等转发间接使用 `pool_store`，但 `__init__.py` 应反映完整的模块清单。

**建议：** 加上 `pool_store` 和 `system`。

### 🟢 低优先级（锦上添花）

#### 3.6 多处硬编码 `PROJECT_ROOT` 路径计算

三个模块各自独立计算项目根路径，都依赖 `Path(__file__).resolve().parent.parent.parent` 的三层回溯：

| 文件 | 变量名 |
|------|--------|
| `rquant/business/data.py:19` | `DATA_DIR` |
| `rquant/business/pool_store.py:23` | `_PROJECT_ROOT` |
| `rquant/research/workflow.py:25` | `PROJECT_ROOT` |
| `rquant/data_source/eastmoney.py:21` | `_DB_PATH` |

路径计算逻辑分散且脆弱——如果目录层级变化（如 `rquant/business/` 再嵌套一层），所有硬编码的 `.parent.parent.parent` 都需要调整。

**建议：** 在 `rquant/__init__.py` 或 `rquant/data_source/cache.py` 中定义统一的 `PROJECT_ROOT` 常量，各模块从一处引用。

#### 3.7 `eastmoney.py` 自建 SQLite 连接而不复用 `db.py`

`eastmoney.py` 直接 `sqlite3.connect()` 创建独立连接，不经过 `db.py` 的 thread-local 连接管理。虽然 `eastmoney.db` 是独立数据库文件，但连接管理、WAL 模式、busy_timeout 等最佳实践需要重复实现。

**建议：** 为 `db.py` 增加"多数据库"支持（如 `get_conn(db_name="rquant")` / `get_conn(db_name="eastmoney")`），或至少让 `eastmoney.py` 复用相同的连接配置模式。

#### 3.8 `data.py` 内存快照与 `quote_cache.py` 两层缓存

`rquant/business/data.py` 的 `_stock_store`（5 分钟 TTL）和 `rquant/data_source/quote_cache.py` 的 `QuoteCache`（30 秒 TTL）形成两层行情缓存，TTL 不同且职责有重叠。

**建议：** 明确分工——`quote_cache.py` 负责"网络层缓存"（减少 Sina 请求），`data.py` 的 `_stock_store` 改为仅维护已查看过的股票元信息（name/sector），不重复缓存行情数据。

#### 3.9 `workflow.py` 的 `pyarrow` 依赖未声明

验证时 `research.workflow` 因缺少 `pyarrow` 模块导入失败。`parquet_store.py` 的 `import pyarrow` 在 `workflow.py → load_kline() → parquet_store.read()` 链路中被触发，但 `pyarrow` 不是所有场景都必需（仅 Parquet 路径需要）。

**建议：** `parquet_store.py` 的 `import pyarrow` 改为延迟导入（在 `read()` / `write()` 函数内部），或至少在 `requirements.txt` / `pyproject.toml` 中标注为可选依赖。

## 四、耦合度分析

```
策略层 ← → 研究/回测层 ← → 业务层 ← → 数据层
  │                                │
  └──────── Web 层 ────────────────┘
```

| 依赖方向 | 耦合方式 | 评价 |
|----------|----------|------|
| business → data_source | 通过 `rquant.data_source.pool` 全局单例 | ✅ 合理 |
| web → business | 通过函数调用 | ✅ 合理 |
| web → strategy | 通过 `scan_stock()` 等 API | ✅ 合理 |
| research → strategy | workflow.py 直接 import MultiFactor 类 | ⚠️ 绕过了注册中心 |
| research → business | 通过 `rquant.business.data` | ✅ 合理 |
| backtest → strategy | 通过 `Strategy` Protocol（类型标注） | ✅ 合理 |
| data_source.pool → data_source.sina | 模块级硬编码导入 | ⚠️ 解耦不彻底 |

没有发现循环依赖，跨层调用方向符合分层架构预期。

## 五、冗余分析

| 位置 | 问题 | 严重程度 |
|------|------|----------|
| `data.py` `_stock_store` ↔ `quote_cache.py` | 两层行情缓存，TTL 不同（300s vs 30s） | 低 |
| `eastmoney.py` SQLite 连接管理 | 与 `db.py` 重复实现连接逻辑 | 低 |
| 多处 `parent.parent.parent` | 路径计算逻辑重复 4 次 | 低 |
| `portfolio.py` vs `data.py` 的 `_save_json` | `portfolio.py` 已正确复用 `data.py` 的工具函数 | ✅ 无冗余 |

**整体冗余度较低。** 之前的重构（合并 ETF 字典、合并镜像函数）已经消除了一批重复代码。

## 六、过分设计评估

| 模块 | 评估 |
|------|------|
| `rquant/data_source/mq.py` | 156 行的简易内存 pub-sub 消息队列。设计克制，没有引入 RabbitMQ/Kafka 等重量级依赖。当前用途明确（异步刷盘 + 行情通知），没有过分设计。 |
| `rquant/backtest/engine.py` | 594 行，包含完整的 A 股交易约束建模。功能完整但不过度——如果只做简单回测可以更薄，但当前设计服务于"策略对比链路"，复杂度和使用场景匹配。 |
| `rquant/research/workflow.py` | 700 行，是项目最长的单文件之一。`select_liquid_board_pool()` 内部嵌套了多级循环和指标计算逻辑，可考虑拆分为 `pool_selection.py` + `stock_metrics.py`。 |

**结论：没有明显的过分设计。** 各模块的行数和复杂度与职责匹配。`workflow.py` 稍长，但属于编排逻辑的自然聚集。

| 优先级 | 问题 | 改动范围 | 风险 |
