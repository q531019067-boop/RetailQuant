# 技术文档写作指南

根据实战经验总结的技术文档写作规范。适用于坐标系转换、接口文档、系统架构等技术参考文档。

## 核心原则

1. **场景驱动** — 按"我要做什么"组织，不按"系统分类"罗列
2. **速查优先** — 开头放总表，读者 3 秒找到答案
3. **不用行号** — 用文件路径 + 函数名（行号会变，函数名不会）
4. **纵向流程图** — mermaid 用 `flowchart TD`，不用 `LR`（横向太挤）
5. **自包含** — 每个场景独立成节，读者跳着看也能看懂
6. **易错点单独列** — 把踩过的坑集中放在一个醒目的章节
7. **总览先行** — 架构文档开头放全局流程图 + 核心设计决策，30 秒建立心智模型
8. **先答核心问题** — 读者最关心的问题（是否可逆？怎么检测？）放在章节最前面
9. **图小字大** — 一个图只表达一个判断，配合文字解释，不要 20+ 节点大图
