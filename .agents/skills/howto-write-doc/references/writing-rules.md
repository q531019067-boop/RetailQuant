# 写作规范

## 规则清单

### 1. 不用行号

行号会随代码修改而变化。用文件路径 + 函数名定位。

```
好的：KSkill.cpp 函数 CastOnPointSector
坏的：KSkill.cpp:4485
好的：kmathtools.cpp 函数 YawRHToLH
坏的：kmathtools.cpp:467
```

### 2. 流程图用纵向

mermaid 用 `flowchart TD`（上到下），不用 `LR`（左到右）。横向太挤，节点多时完全看不清。

### 3. 速查总表放最前面

读者进来第一眼应该看到总表。总表要求：
- 一行一个转换/公式
- 覆盖所有方向（A→B 和 B→A）
- 不需要看后面就能用

### 4. 场景节自包含

每个场景节要能独立阅读。包含：
- 引用块一句话说明场景
- 流程图
- 完整代码片段（可复制）
- 接口位置（文件路径 + 函数名）

### 5. 易错点单独成章

把踩过的坑集中放在文档末尾（常量表之前）。每个坑：
- 标题 = 坑的名称
- 先说现象（用错会怎样）
- 再说正确做法
- 有对比就用代码块对比

### 6. 不写反面教材的长篇解释

一句话说清楚"不要用 X，用 Y"就够了。不需要解释 X 为什么错。

### 7. 公式要可复制

公式用代码块包裹，不要用 LaTeX（在 markdown 渲染器中可能不支持）。

### 8. 常量表放最底部

常量是参考信息，不是核心内容。放在文档最后。

### 9. 产品库文档不写 C++ 代码

产品库文档的读者是策划和测试，他们看不到 C++ 源码。遇到 C++ 变量时，必须追溯到配置源头：

**tab 表变量**：写清楚文件路径、列名、值格式，同时保留公式说明用法
```
好的：
配置文件: client/settings/NpcTemplate/<副本目录>/sNpcTemplate.tab
列名: NpcSeasonDamageScale
值格式: 千分比，1024 = 100%，0 或空 = 不缩放
公式: 最终伤害 = 原始伤害 × nNpcSeasonDamageScale / 1024

坏的：
pBullet->nNpcSeasonDamageScale = pNpc->m_pTemplate->nNpcSeasonDamageScale;
```

**脚本变量**：写清楚配置方式、路径、写法
```
好的：
配置文件: client/scripts/skill/纯阳/天道剑势_万剑归宗.lua
配置方式: 在 GetSkillLevelData 函数中设置 skill.nAngleRange = 256

坏的：
skill.nAngleRange = 256; -- 施法角度
```

### 10. 流程图中不同时机的流程不要用实线连接

如果两个流程发生在不同时间（如"挂载时"和"每跳执行"），不要用 `-->` 直接连接。用虚线 `-.->` 并标注触发时机。

```
好的：
A["挂载时快照"] -.->|"每跳触发"| B["每跳执行"]

坏的：
A["挂载时快照"] --> B["每跳执行"]
```

实线 `-->` 表示连续发生的流程。虚线 `-.->` 表示有时间间隔或事件触发。

## 检查清单

写完文档后检查：

- [ ] 开头有速查总表？
- [ ] 每个场景有流程图？（mermaid TD）
- [ ] 流程图是纵向的？（不是 LR）
- [ ] 没有行号？（用文件名+函数名）
- [ ] 代码片段可复制？
- [ ] 易错点单独成章？
- [ ] 常量表在底部？
- [ ] 没有孤岛章节？（每个章节在总表中有对应）
- [ ] 产品库文档：C++ 变量追溯到了 tab/脚本配置？
- [ ] 产品库文档：tab 表写了文件路径+列名+值格式？
- [ ] 流程图：不同时机的流程用虚线连接？
