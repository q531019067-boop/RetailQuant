# 文档结构模板

## 场景驱动的标准结构

```
# 文档标题

## 速查总表          ← 开头放一张表，覆盖所有转换/公式/接口
| 从→到 | 方法 | 一句话公式 |

## 场景一：xxx       ← 按"我要做什么"组织
> 一句话说明场景

流程图（mermaid TD）
代码片段（完整可复制）
接口位置（文件路径 + 函数名）

## 场景二：xxx
...

## 易错点            ← 集中列出踩过的坑
### 1. 坑的名称
问题描述 + 正确做法 + 错误做法

## 常量速查          ← 底部放常量表
```

## 为什么这样组织

| 传统写法 | 场景驱动写法 |
|---------|------------|
| 按系统分章（引擎、逻辑、屏幕） | 按场景分章（画到屏幕上、转游戏坐标） |
| 读者要从头读到尾拼接转换路径 | 读者直接跳到自己的场景 |
| 公式散落在各章节 | 速查总表一目了然 |
| 行号引用（容易过期） | 文件名+函数名（稳定） |
| 横向流程图（挤） | 纵向流程图（清晰） |

## 速查总表的设计

总表是文档最重要的部分。要求：

- **一行一个转换** — 从 A 到 B，用什么方法，一句话公式
- **覆盖所有方向** — 不要只写 A→B，也要写 B→A
- **放在文档最前面** — 读者进来第一眼看到
- **不需要看后面的内容就能用** — 总表自包含

## 场景节的设计

每个场景节包含：

1. **引用块一句话说明** — `> 我有游戏坐标，想画到屏幕上`
2. **mermaid 流程图** — 用 `flowchart TD`（纵向），不用 `LR`（横向太挤）
3. **代码片段** — 完整可复制，不要省略关键步骤
4. **接口位置** — 文件路径 + 函数名，不用行号

## 易错点节的设计

- 每个坑一个子节，标题就是坑的名称
- 先说现象（用错会怎样），再说正确做法
- 如果有正确/错误对比，用代码块对比展示
- 放在文档末尾（常量表之前），因为读者通常是遇到问题才来查
