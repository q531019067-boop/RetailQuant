# 架构深潜文档写作规范

适用于系统架构、引擎内部机制、运行时流程等"深度理解"类文档。与场景驱动文档不同，架构文档的读者需要理解系统如何运作，而不是查找某个公式或接口。

完整示例（含每条规则的 before/after 对比）：`examples/architecture-deep-dive.md`

## 规则清单

### 1. 总览先行，由浅入深

文档开头必须是全局总览（一张流程图 + 核心设计决策），让读者 30 秒建立心智模型。设计思想放在总览里，不要放到最后当"总结"。每个设计决策 1-2 段，讲 why 不讲 how。

### 2. 每个章节有要点，每个代码块有目的

`##` 章节前加 `> **本章要点：**` 一句话，读者决定是否深入。代码块前加 `> **核实依据：**` 或 `> **结构定义：**` 一句话，读者看总结就懂，代码只是核实用。

如果读者需要看代码才能理解你在说什么，说明文字总结不够。

### 3. 先回答读者最关心的问题

不要上来就讲触发阈值和常量定义。读者关心本质问题（是否可逆？怎么检测？谁来触发？），先用 Q&A 直接回答，再展开细节。

### 4. 讲概念时也要讲实现方式

不能只写 what 不写 how。列出关键状态变更路径，写明触发方式（工具调用？命令？自动？）、校验逻辑、失败处理。违反直觉的实现方式（如 goal 完成只能通过工具调用，模型文本不被识别）本身就是最重要的知识点，必须显式说明。

### 5. 有损/不可逆性质必须开门见山

如果某个机制是有损的，在章节开头就声明，配对比表（保护什么 / 不能保护什么）。不要让读者读到一半才发现"这是不可逆的？"

### 6. 违反直觉的声称必须独立核实

当文档中出现违反直觉的内容，不要因为代码确实如此就照搬。用独立 agent 从不同角度核实：有没有间接恢复路径？排序是否导致分支很少被触发？在文档中注明核实的角度和为什么结论仍然成立。

### 7. Mermaid 图要小而精，配合文字解释

一个图只表达一个判断分支，不要把所有路径塞进一张大图（不超过 10 个节点）。图旁边必须有文字解释——图是概览，文字是详解。先文字逐层说清，再放小图总结。

### 8. What → Why → How 的顺序

先讲是什么，再讲为什么这么设计，最后讲代码怎么实现的。不要上来就堆代码和常量定义，读者还不知道这是在解决什么问题。

### 9. 流程图要用代码证明

mermaid 图里的每个节点和分支都要有对应代码引用。如果代码里没有某个路径，图里就不应该画。反过来，如果代码里有重要分支图里没画，就是遗漏。

### 10. 表格比段落更适合对比和枚举

保护范围、触发条件、快照时机——3 组以上对比信息用表格，不要用段落揉在一起。

## 检查清单

写完架构文档后检查：

- [ ] 开头有全局总览（一张图 + 核心设计决策）？
- [ ] 每个章节有 `> **本章要点：**`？
- [ ] 每个代码块有 `> **核实依据：**` 或类似的一句话总结？
- [ ] 读者最关心的问题在章节最前面直接回答了？
- [ ] 有损/不可逆性质开门见山声明了？（有对比表？）
- [ ] 违反直觉的声称有独立核实？
- [ ] mermaid 图小而精（≤10 节点）？
- [ ] 图旁边有文字解释？
- [ ] What → Why → How 顺序正确？
- [ ] 流程图每个节点都有代码对应？
- [ ] 对比信息用表格而不是段落？
